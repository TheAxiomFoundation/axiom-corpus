Signed the `releases/r0/release_manifest.json` corpus release manifest in place, now that `AXIOM_CORPUS_RELEASE_SIGNING_KEY` is provisioned. Added `axiom-corpus-release sign-release-manifest`, which attaches (or replaces) the HMAC-SHA256 signature on an already-emitted manifest without rebuilding it: the signature is computed over `canonical_manifest_bytes` (which excludes the `signature` field), so every hashed field — artifact SHA-256s, row counts, `created_at`, and `git` — is left byte-identical. This is the correct path for r0, whose content is pinned to commit `9ee50de` (its branch point); re-emitting at the current `HEAD` would re-stamp `created_at`/`git` and re-hash the moved tree, redefining the release rather than signing it. The command guards that canonical content is unchanged before writing, is idempotent (reports `already_valid` when re-signed with the same key), and requires the key. `verify-release-manifest` passes with the key and fails with a wrong key.
