Make Supabase provision staging idempotent against verified pre-staged state. `load_provisions_to_supabase` now fetches every loaded scope's existing rows before writing: byte-identical rows are no-ops, rows whose content matches but whose derived `id`/`parent_id` came from a superseded id scheme are converged to the canonical projection (planned against the `parent_id` ON DELETE CASCADE closure so no surviving row is ever silently deleted, with the dependent set re-checked immediately before deletes), and divergent content under the same immutable `(citation_path, version)` key — or staged rows the load does not describe — raises `ProvisionStagingConflictError` before any write. The `identifiers` jsonb column compares by canonical JSON so a boolean-vs-numeric value change cannot pass as equal; inserts are ordered by a topological sort over actual parent links (cyclic or id-colliding projections are conflicts, not runtime constraint errors); in-place converges write only `parent_id`, conditioned on the verified prior value. The previous `on_conflict=id` merge upsert, which silently overwrote same-id rows and crashed with a 23505 duplicate-key error when an earlier ingest had staged the same citations under different ids, is removed. `publish_corpus.py` now stages all of a release's unreleased scopes in one call so stale parent links crossing scope boundaries within a release converge together, and the int4 ordinal shim indexes within each scope so a multi-scope load projects exactly the per-scope signed evidence digests. Publishing a release over pre-staged identical provisions (the et/zm/ug/gh SOUTHMOD failure) now succeeds instead of dying at staging.
